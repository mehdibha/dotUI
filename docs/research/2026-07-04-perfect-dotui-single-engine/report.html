The Perfect dotUI

The definitive end-state architecture for a design-system builder in which every visual decision is a configurable axis, the preview is the export, and one document compiles to idiomatic Tailwind v4 — styles are Tailwind, resolved not lifted.

How this study was produced. Derived by transforming the two-engine end-state study for a single-engine world: the owner ruled the second engine out of scope, which removes the forcing function behind the intermediate style IR. Chapter 01 01-system-overview.md

One document. Two pure functions. Two identical execution sites.

Everything the system does is a projection of a dsdoc through resolve() then compile(). The builder's preview and the export endpoint run the same compiled bundle — so fidelity isn't tested into place, it's a property of there being only one compiler.

  • The dsdoc is the only thing the user owns and edits. The Registry Manifest is dotUI's frozen vocabulary — an immutable snapshot the dsdoc pins by lock. An omitted value means the pinned manifest's default, never "whatever today's code says." That single rule kills silent-reinterpretation for good.
  • Component styles are authored as Tailwind tv() strings and resolved — params, density, scalar vars, declared var-writes — into the shipped tv() file. There is no intermediate IR: styles are Tailwind. The class string on a preview node is character-for-character the class string in the exported file.
  • Every value flows through the Dimensional Token Graph: three layers, stable readable ids, mode dimensions × cells. A retarget or a hue drag is a CSS-variable write — zero re-renders.
  • Three version numbers never collide: dsdoc (schema shape → migration ladder), meta.version (the user's content), lock.manifest (the pinned vocabulary).

Chapter 04 04-styles.md

Author in Tailwind. Resolve, don't lift. Emit idiomatic tv().

Contributors write Tailwind tv() strings in styles.ts. The compiler resolves them — params, density, scalar vars, declared var-writes — into the shipped tv() file. No lift, no intermediate IR: styles are Tailwind, with the full utility surface (:has(), **:[svg], peer-*, arbitrary values). The one surviving style lint is the hardcoded-value discipline — a warning with a token hint, not a gate.

Worked example — the Button, authored and shipped

Emitted Tailwind — button.tsx byte-comparable to hand-written tv()
const buttonVariants = tv({
  base: [
    "inline-flex items-center justify-center select-none",
    "rounded-(--btn-radius) font-(--btn-weight)",
    "focus-visible:focus-ring",
    "**:[svg]:size-3.5 has-data-icon-end:pr-2", // authored as-is — nothing to re-home
    "disabled:opacity-50 pressed:translate-y-px",
  ],
  variants: {
    variant: {
      // flatten-on-export: default target → semantic utility
      primary: "bg-primary text-fg-on-primary",
      quiet:   "bg-transparent hover:bg-inverse/10",
    },
  },
})
The menu.highlight bug is now unrepresentable. Named-style vars blocks and inline [--color-highlight:…] writes are carried into the shipped tv() verbatim — resolution has no strip step where a var could be dropped. The old failure (vars stripped at publish, so the export silently diverged from the preview) can't recur, and a resolution-completeness test asserts every referenced var survives.
States are plain Tailwind variants over React Aria data-attributes
StateTailwind variant
hover · focusVisiblehover: · focus-visible:
pressed · pending · disabledpressed: · pending: · disabled: (RAC data-attrs)
invalid · openinvalid: · open: (self data-attrs)
iconStart · iconEndhas-data-icon-end: via :has()
inModal · inPopoverin-data-modal: (ancestor context)
There is no dual-binding vocabulary and no runtime children-scan — a state is one variant with one data-attribute behind it. Whether a component exposes free-form children (matched with :has()) or explicit prefix/suffix slots is a component-API choice, not a styling constraint.

Tradeoffs it accepts

  • No stable-id targeting: codeStyle transforms and shadcn diffs key off the tv() structure (base / slot / variant / compound) and named-style ids, not rule-ids on a JSON contract.
  • Delta storage is engine-specific: user overrides are raw Tailwind class strings — moot with one engine, but it forecloses free re-targeting to a second engine later.
  • The hardcoded-value lint is a warning, not a gate — a contributor can justify a literal, so discipline (not the build) carries the last mile.
Chapter 05 05-tokens.md

The Dimensional Token Graph

One graph replaces three disconnected representations. Three layers as enforced edge rules, permanent readable ids, and modes modeled as orthogonal dimensions — so high-contrast composes with every scheme instead of being hand-authored per combination.

Chapter 08 08-density-sizing.md

Density is a dimension, authored as geometry

Three tiers — compact / default / comfortable — are a tv() dimension, not a CSS scale factor. Real height ladders are non-linear per rung, so no single multiplier reproduces them. The sizes() helper authors the ladder as one reviewable table.

Three orthogonal "how big" levers that compose without interference
LeverWhat it isPicksTier
densitysystem-wide axis, role:'density'the active column of the geometry table for every controlstructural
size propper-instance prop on one <Button>the row for that one controlnot an axis
spacing scalesystem-wide axis, value tierredefines what each spacing.N token meansvalue
Export offers codeStyle.density: 'baked' (default — fold the selected tier into classes, leanest) or 'runtime' (ship the full data-density axis, ~3× the geometry classes, switchable in the consumer's app). dotUI's tiers map 1:1 to shadcn's style-mira / style-nova / style-vega.

Chapter 10 10-builder.md

The entire panel is generated from AxisDecl declarations — one declaration drives the control, the ⌘K index, the AI vocabulary, and the tier classification. The toCommand signature makes a stub control a type error, so unconsumed knobs are unrepresentable.

Chapter 13 13-testing.md

Eight invariants, five gates

Fidelity is mostly structural — the same compile() runs in both realms — so the suites are narrow, targeting only the seams the structure can't close. Each invariant protects exactly one product promise and runs at the earliest gate that can catch its failure. With one engine there is no cross-engine matrix to prove agreement over.

Seven recipes, one review question

Every contribution lands in registry source or in machinery, and the distinction governs everything. The recurring question, applied at the right layer: would two design systems disagree on this? If yes, it's an axis. If it's mechanics, it stays a plain value. If a look has no covering axis, it's flagged — never patched with an invented token.

Governance policies

  • Hardcoded-value discipline: a design-meaningful literal (bg-[#635bff], rounded-[7px]) is a dotui/hardcoded-value warning carrying a token hint — a contributor can justify it; component mechanics (p-0, hairlines, -translate-x-1/2) stay plain values. A warning, never a build gate.
  • Id permanence: every referent has a permanent readable id; labels rename freely, ids never; dotui/id-permanence fails a PR that removes or reuses a published id.
  • Manifest release: a content-addressed snapshot is the release unit — additive changes are minor bumps that move no stored doc; breaking changes are major bumps carrying a deprecation table reconcile() reads.
