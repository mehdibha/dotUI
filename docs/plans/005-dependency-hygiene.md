## Why this matters

`pnpm audit --prod` reports high-severity advisories in transitive deps at planning time: `seroval` (RCE/prototype-pollution advisories, pulled via `@tanstack/react-devtools` → solid-js chain) and `next` 16.0.x–16.2.x (DoS advisories, pulled via `fumadocs-core`/`fumadocs-mdx`). Exploitability in this architecture is low (devtools are dev-tooling; the site doesn't run Next), but high-severity noise in the audit hides real future findings and the fixes are within-range updates. Separately, `puppeteer` (~150 MB of install weight) sits in root devDependencies with **zero imports anywhere in the repo** (verified by grep), and the `tailwindcss-react-aria-components@2.1.1` patch may be obsolete if upstream shipped the `not-*` variant fix.

## Steps

### Step 1: Record the baseline

`pnpm audit --prod 2>&1 | tail -20` — record the totals (planning-time observation was ~35 advisories incl. ~17 high; numbers will have drifted). `du -sh node_modules/.pnpm/puppeteer*` for the before-size if present.

**Verify**: you have the numbers saved for the final report.

### Step 2: Refresh the advisory chains

Run `pnpm update fumadocs-core fumadocs-mdx @tanstack/react-devtools @tanstack/react-router-devtools --filter www` (within existing ranges), then `pnpm audit --prod` again.

- If the `seroval` and `next` high-severity advisories clear: done.
- If they remain because a parent pins an old transitive: add the minimal `overrides` entry in `pnpm-workspace.yaml` (e.g. `seroval: '>=1.4.1'`) — prefer the narrowest override that clears the advisory, and add a one-line comment with the advisory id and a removal condition.
- If clearing them requires a MAJOR bump of fumadocs or devtools: STOP (out of scope) and report which majors would be needed.

**Verify**: `pnpm audit --prod` shows zero high-severity advisories in the `seroval` and `next` chains (other/new advisories: record, don't chase). `pnpm install` exits 0.

### Step 3: Drop puppeteer

Remove `"puppeteer": "^25.0.2"` from root `package.json` devDependencies and the `puppeteer: true` line from `allowBuilds` in `pnpm-workspace.yaml`. Run `pnpm install`.

First re-verify it is truly unused: `grep -rn "puppeteer" --include="*.ts" --include="*.tsx" --include="*.mts" --include="*.mjs" --include="*.json" www packages .github package.json | grep -v pnpm-lock` → only the package.json line you are deleting (and possibly this plans directory).

**Verify**: `pnpm install` exit 0; the grep above (after the edit) → no matches outside `docs/plans/` and the lockfile.

### Step 4: Check the RAC patch upstream (report-only)

Read `patches/tailwindcss-react-aria-components@2.1.1.patch` to understand what it fixes. Check whether `tailwindcss-react-aria-components` has releases newer than 2.1.1 (`pnpm view tailwindcss-react-aria-components versions --json | tail -5`) and, if network access allows, whether their changelog mentions the `not-*`/`:is()` selector fix. Write the verdict (fixed-upstream-in-X / still broken / inconclusive) into your final report and the plans README status row — do NOT change the dependency or the patch.

**Verify**: verdict recorded; `git status` shows no changes under `patches/`.

### Step 5: Full verification sweep

`pnpm check && pnpm typecheck && pnpm test && pnpm build:registry && pnpm build:www`

**Verify**: all exit 0; after `build:registry`, `git status` shows no modified generated artifacts (dep refresh didn't alter codegen output — if it did, inspect the diff: formatting-tool or shiki updates can legitimately change generated output; commit it if so and say so in the report).
