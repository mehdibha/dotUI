## Direction findings (options for the maintainer — not ranked against bugs)

1. **Preset-themed docs** (strongest, north-star aligned): docs pages render component demos in the default theme only; the builder's preset codec + theming machinery (`modules/create/preset/codec.ts`, `/r/{name}?preset=` flow) already round-trips full design systems. Letting `/docs` read a preset (URL param or persisted from `/create`) and theme every demo/playground would deliver "evaluate any component in YOUR design system" — a differentiator no other registry has, and a natural bridge from docs → builder. Effort L (theme context must reach demo iframes/playgrounds + gallery); product-shaping; plan on request.
2. **Component-status unification** (resolves findings 13–14): pick one status taxonomy, store it in registry `meta.ts` (source of truth), generate the gallery's status and a docs-page WIP badge from it, delete the unused `wip` frontmatter or wire it. Needs the operator to pick the taxonomy + badge design first; the mechanical part is ~a day after that.
3. **Full-text docs search**: the ⌘K palette filters page *titles* only (it's a React Aria Autocomplete over the page tree — `search-command.tsx`; works fine as navigation). fumadocs-core ships search infrastructure the repo already depends on; wiring content-level search is S–M. Grounded but not urgent at 61 pages — title search covers most journeys today.
4. **AI-surface completion**: llms.txt/llms-full.txt/per-page .md/copy-page are all live and generated from the docs source (good). Gaps: `/create` and the registry endpoints aren't listed in llms.txt's link sections (the summary prose mentions `/r/{name}`), and there's no MCP server for the registry — the open-in research (`docs/research/open-in/architecture.md`) names machine surfaces as the keystone. Small llms.txt additions are trivial; MCP is M and product-shaping.

## Findings considered and rejected (binding for future audits — don't re-investigate)

- **TOC thumb observer "stale closure / leak / re-render loop"** (two subagents): false. `toc.tsx:99-124` is the standard pattern — each effect's cleanup disconnects the observer instance it created (closure over the local), `useCallback([active])` re-creates at most one observer per anchor change. No loop, no leak.
- **`dynamic-pre.tsx` placeholder "duplicate React keys across suspensions"**: false. `${line}-${count}` keys are rebuilt deterministically every render; same input → same keys, unique within the list.
- **Docs `$.tsx` loader "needs try-catch around getPage/getText"** and **"setResponseHeader could throw"**: defensive-wrapper noise; the router error boundary is the designed handler; no evidenced failure mode.
- **`code-template.ts` "empty body produces malformed return"**: unreachable — templates are built from real demo sources (always non-empty JSX); the engine is spec-covered (`source-overlay.spec.ts`).
- **`wordPresent` non-ASCII identifier edge** and **`locateAttr` regex lookahead edge**: repo-authored ASCII demo sources; cosmetic tree-shaking at worst; oxfmt normalizes the syntax the regexes see.
- **`process-controls.ts` / `source-overlay.ts` `as`-cast mutations**: type-hygiene style point, no observable bug; the icon-control skip is explicit.
- **"Icon control values could be objects → `[object Object]`"**: no path — values originate from the typed controls UI.
- **"Validate duplicate control names" as standalone bug**: real but tiny — folded into plan 002 Step 4.
- **"meta.json entry missing hides a page"**: not a real failure mode for components — `www/content/docs/components/` has NO meta.json; fumadocs auto-includes the folder. Only `(root)/meta.json` enumerates, and it's 3 entries.
- **"Write unit tests for element-to-code"** (test agent): inverted — all 44 playgrounds are SourceFirst; the file is dead-in-practice and plan 005 deletes it instead.
- **"Hand-rolled sidebar/pager/last-update should use fumadocs built-ins"**: by design — the repo consumes fumadocs-core (headless) precisely to own its UI; fumadocs-ui isn't a dependency.
- **"references module split from docs/reference.tsx is incoherent"**: the boundary is fine (data transform vs presentation); a README there is nice-to-have, not debt.
- **"Docs search is navigation-only with unused search state"** (direction agent): wrong — `Command` wraps React Aria `Autocomplete` with a `contains` filter; typing filters page titles. Reframed as direction finding D3 (full-text is the gap, not search itself).
- **"registry.json / export endpoints don't exist"** (direction agent): wrong — `www/src/routes/r/registry[.]json.tsx` exists. The llms.txt gap claim was kept (direction D4) after correction.
- **"Auto-generate playgrounds from extracted prop types"**: dropped by the direction agent itself — controls are deliberately curated; no friction signal.
- **Component gallery eagerly imports all 64 demos**: accepted design — single prerendered gallery route pays its own weight; revisit only if Core Web Vitals on `/docs/components` regress.
- **Demo/Example dual HAST (source+preview) inlined per node**: accepted design — instant toggle UX; explicitly evaluated in the research doc (option 1 kept).
- **Build memory / Vercel machine**: resolved before this audit — see `docs/research/2026-06-11-docs-build-memory.md`; machine downgrade is a dashboard action pending on the operator.
- **`globals-docs` (2019) abandonment**, **ts-morph 28 ↔ TS 6 compat**, **unified/unist/mdast tooling "duplication"**: all cleared — static data with curated overrides; compatible; properly layered single-purpose tools.
- **typescript-api-extractor beta pin**: re-checked 2026-06-12 — still no stable, no newer beta; keep the repo-audit watch item, nothing to do.
