# 001 — Block & layout registry items + variant model

> Read the [folder README](README.md) first (especially Phase 0 decisions and the Architecture paragraph). Planned at `744a9179`. This plan defines the registry shape for blocks/layouts and authors the first block end-to-end. It gates 002–005.

## Goal

Establish blocks/layouts as a new registry item family with a single named `variant` axis that resolves at publish into one canonical file, and author the first block (Login) with ≥2 variants as the reference. No `/create` UI and no `/blocks` page here — just the source-of-truth items and their types.

## What exists (verified)

- A registry item lives in `www/src/registry/ui/<name>/` with `meta.ts`, `base.tsx` (+ optional `base.<variant>.tsx`), `index.tsx`, `types.ts`, `demos/`, `examples.tsx`, optional `styles.ts`/`styles.css`.
- `RegistryItem = ShadcnRegistryItem & { group?: ComponentGroup | null; params?: Record<string, ParamDef> }` in `www/src/registry/types.ts`.
- **The variant precedent — `loader`** (`www/src/registry/ui/loader/meta.ts`, read in full):
  ```ts
  const loaderMeta = {
    name: 'loader',
    type: 'registry:ui',
    group: 'progress',
    files: [{ type: 'registry:ui', path: 'ui/loader/base.spinner.tsx', target: 'ui/loader.tsx' }],
    params: {
      style: {
        kind: 'enum',
        default: 'spinner',
        values: ['spinner', 'ring'] as const,
        files: {
          spinner: [{ type: 'registry:ui', path: 'ui/loader/base.spinner.tsx', target: 'ui/loader.tsx' }],
          ring:    [{ type: 'registry:ui', path: 'ui/loader/base.ring.tsx',    target: 'ui/loader.tsx' }],
        },
      },
    },
  } satisfies RegistryItem
  ```
  `loader` is **styles-less** (no `styles.ts`; the dir holds `base.ring.tsx`, `base.spinner.tsx`). Picking `style: 'ring'` ships `base.ring.tsx` as `ui/loader.tsx`. This is the block-variant model exactly.
- Items register via `www/src/registry/__generated__/registry-items.ts` (`registryUi`, `registryLib`), regenerated by `pnpm build:registry` (`www/scripts/registry-build.ts`) globbing `ui/*/meta.ts`. There is **no** `registryBlocks` array yet — adding it is 002.
- Component imports inside a base file must all appear in `registryDependencies` or `checkRegistryDepsDrift` fails the build.

## Steps

### Step 1 — Add the block item type to the schema

In `www/src/registry/types.ts`:

1. Add a block category union and a block item type alongside `RegistryItem`:
   ```ts
   export type BlockCategory = 'layout' | 'page' | 'section'
   // 'interactive' is reserved for the deferred real-world-blocks work — do NOT add it in v1.

   export type BlockRegistryItem = ShadcnRegistryItem & {
     category: BlockCategory
     // The single tweak. Always present, always named `variant`, always enum-with-files.
     params: { variant: EnumParamDef }
     // Builder/explorer display metadata (named variants, descriptions, blurb).
     display?: {
       title: string
       description: string
       variantLabels?: Record<string, string> // value -> human label ("split" -> "Split")
     }
   }
   ```
   Reuse the existing `EnumParamDef` (the same type `loader.params.style` satisfies). Do **not** invent a second param kind — decision #3 forbids a second knob.
2. Do **not** add a `ComponentGroup` member for blocks. Blocks must not carry `group` (gotcha: keeps them out of the components list / docs gallery).

STOP if `ShadcnRegistryItem`'s `type` union does not accept `'registry:block'` — Agent-mapped as supported, but verify by reading the imported schema; if absent, report rather than widening the vendored schema yourself.

### Step 2 — Create the blocks source directory + the first block

Create `www/src/registry/blocks/login/`:

- `meta.ts`:
  ```ts
  import type { BlockRegistryItem } from '@/registry/types'

  const loginMeta = {
    name: 'login',
    type: 'registry:block',
    category: 'page',
    files: [{ type: 'registry:block', path: 'blocks/login/base.centered.tsx', target: 'login.tsx' }],
    params: {
      variant: {
        kind: 'enum',
        default: 'centered',
        values: ['centered', 'split', 'minimal'] as const,
        files: {
          centered: [{ type: 'registry:block', path: 'blocks/login/base.centered.tsx', target: 'login.tsx' }],
          split:    [{ type: 'registry:block', path: 'blocks/login/base.split.tsx',    target: 'login.tsx' }],
          minimal:  [{ type: 'registry:block', path: 'blocks/login/base.minimal.tsx',  target: 'login.tsx' }],
        },
      },
    },
    registryDependencies: ['button', 'text-field', 'card', 'checkbox' /* every component each variant imports */],
    display: {
      title: 'Login',
      description: 'A sign-in screen.',
      variantLabels: { centered: 'Centered', split: 'Split', minimal: 'Minimal' },
    },
  } satisfies BlockRegistryItem

  export default loginMeta
  ```
  Mirror loader exactly: top-level `files` = the default variant file; `params.variant.files.<each>` = per-variant files; **all share `target: 'login.tsx'`**.
- `base.centered.tsx`, `base.split.tsx`, `base.minimal.tsx`: the three compositions. Each is a complete screen built **only** from `@/registry/ui/*` components and tokens. **No magic numbers for anything a design system would disagree on** (colors, radius, type, density-affected spacing go through the components/tokens); plain structural values (page max-width, grid gaps that are pure layout) are fine — same test CLAUDE.md applies to components.
- `index.tsx`: www-side wrapper if a variant needs router links (e.g. "Forgot password?" `LinkButton`); otherwise re-export. Never shipped.
- `examples.tsx` and/or a preview entry: deferred to 003/004 wiring; a single default export rendering the default variant is enough here.

Author each variant against the **axis-robustness gate** (README Conventions): eyeball it at radius max + density comfortable + flat foundation + a non-default accent before calling it done. This is the real work; the registry wiring is trivial by comparison.

### Step 3 — Decide and document the variant-source convention

Two acceptable layouts for the per-variant files; pick one and document it in the block dir or this plan:

- **(A) One file per variant** (`base.<variant>.tsx`) — matches `loader` exactly, simplest for the publisher, recommended.
- **(B) Shared sub-parts + thin variant files** — if variants share large chunks (a `<LoginForm/>` used by all three), factor the shared part into a co-located file listed in `meta.files` as an extra file, and keep each `base.<variant>.tsx` thin. Only do this if duplication is genuinely large; it complicates `registryDependencies` and extra-file keying (see 002 gotchas).

Default to (A). Revisit (B) per block only when a variant set shares a substantial sub-composition.

## Verification

- `pnpm typecheck` passes (the new types compile; `loginMeta satisfies BlockRegistryItem`).
- The block does **not** yet need `build:registry` to pass — that requires the `registryBlocks` glob from 002. Until 002 lands, the item is source-only and will not appear in any manifest. Note this explicitly when handing off; do not try to force `build:registry` green here.
- Manual: render each `base.<variant>.tsx` in isolation (a scratch route or the existing preview harness once 003 exists) and confirm it composes and survives the axis extremes.

## Done criteria

- `BlockRegistryItem` + `BlockCategory` exist in `registry/types.ts`; no new `ComponentGroup` member.
- `www/src/registry/blocks/login/` exists with `meta.ts` + ≥2 (target 3) variant files, each a token-only composition that passes the axis-robustness eyeball test.
- `registryDependencies` lists every `@/registry/ui/*` imported across all variant files.
- The variant-source convention is documented.

## STOP conditions

- If the vendored shadcn schema does not include `'registry:block'`, STOP and report (do not edit vendored schema).
- If authoring a variant tempts you toward a new token or a second param, STOP — that is a Phase 0 violation (#2/#3). Flag the missing axis to the maintainer instead.
- Do not build the `/create` section or `/blocks` page here (003/004). Do not edit `registry-items.ts` by hand (002 generates it).
